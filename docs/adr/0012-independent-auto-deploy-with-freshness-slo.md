# ADR-0012: 独立自动发布与时间型温备新鲜度

个人博客的内容由 Git 和异地镜像保证零数据丢失，短时间展示旧内容不等同于数据丢失。为避免温备平台故障阻塞健康的主站，`main` 表达两版共同的生产意图：Pull Request 必须同时通过两版构建和内容检查，合并后 Vercel 与 Pages 独立自动部署，不设置日常人工 Production promotion，也不要求 Pages 成功后才能发布 Vercel。

“同时发布”表示由同一合并事件触发，不表示两个平台原子地同时完成。两版线上提交应在正常情况下于合并后十五分钟内收敛到最新可发布的 `main` 提交；超过十五分钟即告警并重试最新提交。该目标不承诺“最多落后一个版本”，因为在 Pages 故障期间继续发布主站会产生多个中间提交，而阻止这些发布会重新造成反向耦合。

## Consequences

- `main` 是始终可发布的生产意图分支，合并后自动触发两条独立部署链。
- Vercel 与 Pages 可以先后完成，短暂版本差异是正常部署状态，不是数据丢失。
- Pages 失败不阻塞 Vercel；系统告警并直接重试最新提交，不要求逐个补发中间版本。
- 内容数据恢复点目标为零，依靠 Git 历史和异地镜像；Pages 的指标改为十五分钟部署新鲜度目标。
- 人工域名切换恢复时间目标仍为一小时；切换前必须核对 Pages 当前线上提交，允许在故障期间短暂提供较旧但完整的内容。
- 高风险架构迁移可以临时增加人工环境批准，普通文章和项目更新不需要。

## Alternatives Considered

**Pages 先部署、Vercel 后人工提升**

- 未采用：能够保证线上版本严格一致，但会让 Pages 故障阻塞主站发布，并给个人博客的每次内容更新增加不成比例的操作成本。

**承诺 Pages 最多落后一个版本**

- 未采用：若不阻塞后续主站发布就无法保证；使用十五分钟收敛时间比版本计数更准确、可监控。

**完全不检查两版线上差异**

- 未采用：会使温备长期失效而无人知晓，无法支持一小时人工恢复目标。
