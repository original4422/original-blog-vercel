# ADR-0007: 单一 monorepo、两套独立应用与 Pages 温备

双仓库适合验证两条部署路线，但在全部公开语义内容必须统一、Pages 需要成为温备站后，跨仓库复制、权限和同步延迟会成为长期负担。项目改为单一 monorepo，在其中保留 Vercel 与 Pages 两套独立应用，并共享权威内容和内容 schema；两版分别构建、测试和部署，不通过平台条件分支合并成一个应用。

Vercel 继续承载唯一主站，Pages 从同一生产意图独立构建并作为不参与搜索竞争的温备站。两边的线上版本允许在部署窗口内短暂不同，Pages 需要按 ADR-0012 的时间目标收敛。每次可恢复发布需要保留源提交和最后一次已知良好的静态产物。完整 Git 历史还应镜像到 GitHub 故障域之外；该镜像只用于恢复，不成为第二内容源。

## Consequences

- 内容、schema 与跨版本修改可以在一个提交和一个 PR 中原子完成。
- Vercel 和 Pages 保持各自的路由、图片、搜索、动画、构建配置和部署生命周期。
- 任一共享变更必须同时通过两版构建与内容一致性检查后才能合并。
- 单个错误提交可能同时影响两版，因此 Pages 部署需要保留最后一次已知良好版本；故障切换前应核对其线上提交和健康状态。
- 单一 GitHub 仓库形成源码平台故障域，因此使用异地 Git 镜像保护源码和历史。
- 现有两个仓库的承接关系、历史迁移和最终命名另行决策。

## Alternatives Considered

**维持两个源码仓库并同步派生内容快照**

- 未采用：能够保留 Pages 的内容快照，但需要长期维护跨仓库权限、转换提交和一致性状态；两个仓库仍位于同一 GitHub 故障域。

**一个应用内使用平台条件分支**

- 未采用：会让静态导出限制污染 Vercel 原生实现，也会模糊两版的测试和发布边界。

**增加独立内容仓库**

- 未采用：可以中立托管内容，但会增加第三个仓库和协调链，且 monorepo 已能提供原子的一致性。
