# GitHub + Vercel 版本：关键决策记录

本文记录会影响架构、内容维护、部署或视觉一致性的决定。占位内容可直接替换，
但改变这里的架构决策前应同步更新本文。

## D-001：以已核验的上游提交为复现基线

- **决定**：以 `yunshenwuchuxun/yunshen-blogv1` 的提交
  `dfbd51a88ea04423c9c2e13ee83e9eec8b44e480` 为唯一参考源码版本。
- **理由**：锁定提交可以让设计复现可追溯，避免上游后续变化使两个本地版本
  产生无法解释的差异。
- **影响**：保留通用 WebGL 流体、Lenis、Motion/GSAP、主题和 MDX 结构；
  LICENSE 原样保留并在 README 明确归属。

## D-002：复用体验层，不复制个人内容层

- **决定**：复用页面骨架与通用动效实现；移除上游文章、论文、真实项目、头像、
  联系方式和截图，替换为 `original` 与可辨识的示例内容。
- **理由**：MIT 允许代码改造，但个人文字与媒体并不是新站点的身份；把二者分离
  既保持参考站体验，也避免错误归属。
- **影响**：示例 MDX 和自制 SVG 是可运行的内容模板，不宣称是真实经历。

## D-003：Vercel 版保持 Next.js 原生运行时

- **决定**：不做静态导出；搜索索引与 RSS 使用 Route Handler，图片继续使用
  `next/image`，生产环境启用 Vercel Analytics 与 Speed Insights。
- **理由**：这是 GitHub + Vercel 方案与 GitHub Pages 方案的核心差异，也最接近
  上游部署模型。
- **影响**：部署目标必须提供 Next.js 运行时；Vercel 无需数据库或额外 API Key。

## D-004：集中管理站点身份与生产 URL

- **决定**：作者、描述、邮箱、GitHub 和站点 URL 全部由 `site.config.ts` 管理；
  URL 优先级为 `NEXT_PUBLIC_SITE_URL` → `VERCEL_PROJECT_PRODUCTION_URL` →
  `http://localhost:3000`。
- **理由**：上游域名散落在 metadata、sitemap、robots 和文章页，曾出现旧域名
  不一致；单一来源可避免 canonical、RSS 和分享链接再次漂移。
- **影响**：绑定自定义域名时只需设置一个环境变量。HTML 语言固定为 `zh-CN`，
  修复参考源码的语言标记不匹配。

## D-005：内容使用文件型 MDX，不引入 CMS 或数据库

- **决定**：首版文章继续存放在 `app/blog/posts/*.mdx`，构建时读取元数据；搜索
  API 只暴露标题、摘要、标签、日期与 slug。
- **理由**：符合个人博客的小规模维护方式，部署不需要密钥，Git 历史也天然成为
  内容版本记录。
- **影响**：发布文章需要提交代码；未来内容量或多人协作需求显著增加时再评估 CMS。

## D-006：占位媒体统一使用自制 SVG

- **决定**：首页、文章与项目的示例媒体均为本仓库生成的抽象 SVG，不使用上游
  截图或第三方照片。
- **理由**：SVG 体积小、清晰、易于用户后续替换，并能用统一的洋红点缀色保持
  高完成度。
- **影响**：真实内容上线前应替换文字和图片，但占位资源本身可以安全部署。

## D-007：保留参考站的黑白反差与洋红强调

- **决定**：采用黑/白高反差舞台、`#de1d8d` 主色、Merriweather 展示字体与
  Mukta 正文字体；不另造第二套视觉语言。
- **理由**：用户要求与参考站对齐，最可识别的设计特征正是排版、反差色块、流体
  首屏和 sticky 项目预览的组合。
- **影响**：新增组件必须优先复用现有色彩变量、边框密度和动效节奏。

## D-008：动效必须有可访问性降级

- **决定**：尊重 `prefers-reduced-motion`，关键导航保持键盘可达，装饰图形不阻断
  内容读取；WebGL 不可用时仍显示静态渐变背景。
- **理由**：高保真不应以可访问性或故障恢复为代价。
- **影响**：动画属于渐进增强，核心导航、文章和项目内容不依赖动画才能访问。

## D-009：图标是表现元素，不嵌套交互元素

- **决定**：被链接或按钮包裹的图标只渲染 `span`/`svg`，可点击语义由最外层的
  `a` 或 `button` 独占。
- **理由**：上游部分图标自身返回 `button`，在导航中形成 `a > button` 或
  `button > button`，会造成非法 HTML、hydration 与键盘焦点问题。
- **影响**：图标仍可响应父元素 hover，但页面中不出现嵌套交互控件。

## D-010：动态标签参数保持原始值

- **决定**：`generateStaticParams()` 返回原始标签，不提前执行
  `encodeURIComponent()`；仅在生成 sitemap URL 时编码。
- **理由**：Next.js 会负责路径编码。上游的双重编码会让包含空格、中文或 `C++`
  的标签在生产环境中 404。
- **影响**：路由验证必须覆盖 `System Design`、`数字花园` 与 `C++`。

## D-011：发布前以零已知生产依赖漏洞为基线

- **决定**：统一升级到 Next.js `16.2.10`，并通过 pnpm overrides 固定已修复的
  PostCSS `8.5.16` 与 YAML `2.9.0`；每次发布前执行 `pnpm audit --prod`。
- **理由**：初始锁定版本已落入多个高危公告区间，且 Next.js 的精确传递依赖会
  保留旧版 PostCSS。显式覆盖能让构建与审计使用同一组安全版本。
- **影响**：升级依赖后必须重新完成代码检查、类型检查、生产构建和浏览器回归；
  当前审计结果为零已知漏洞。

## D-012：生产 Node.js 固定在 24.x

- **决定**：`package.json#engines.node` 固定为 `24.x`，与本地部署 CLI 使用的版本一致。
- **理由**：宽泛的 `>=20.9.0` 会让 Vercel 在新的 Node.js 大版本发布时自动升级，
  增加未经验证的运行时变化；固定当前 LTS 线能保持构建可复现。
- **影响**：未来升级 Node.js 大版本时，应显式修改该约束并重跑完整验收。

## D-013：忽略生成物时不得屏蔽 App Router 路由目录

- **决定**：只按明确路径忽略 `public/` 下的生成文件，不使用全局 `*.xml` 规则。
- **理由**：App Router 的 RSS 实现位于 `app/feed.xml/route.ts`；全局扩展名规则会把
  整个路由目录排除出 Git，导致本地/手工部署正常而 Git 自动部署返回 404。
- **影响**：部署验收必须在 Git 自动部署完成后重新检查 `/feed.xml`，不能只依赖
  本地构建或 Vercel CLI 上传结果。
